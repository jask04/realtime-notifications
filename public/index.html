realtime-notifications

Real-time notification service — WebSocket delivery, BullMQ jobs, Postgres history. Built with production patterns: idempotency, dead-letter queue, per-recipient rate limiting, graceful shutdown.

Fastify TypeScript Socket.io BullMQ Prisma Postgres Redis

Live demo

Click Connect. We mint a temporary token on the server, open a WebSocket, and subscribe you to your own notification stream.

Idle.

Send yourself a notification. The HTTP request goes to POST /notifications; a worker picks it up off the queue and pushes it back here over the WebSocket.

Demo tokens last 1 hour. Rate-limited to 5 tokens / hour / IP and 10 sends / minute / recipient.

What it does

  • POST /notifications — create a notification, write to Postgres, enqueue a delivery job. Idempotent on a client key.
  • POST /notifications/fanout — same, for up to 1000 recipients in one request. One job per recipient so retries scope properly.
  • WebSocket clients connect with a JWT and receive their notifications live (Socket.io with Redis adapter for horizontal scale).
  • Email worker delivers anything that didn't go via WebSocket — SMTP through Mailtrap in dev, swap for SES/SendGrid in prod.
  • GET /notifications — paginated history of the caller's own notifications, filterable by status and channel.
  • GET /admin/queue/* — operator endpoints for queue stats, DLQ inspection, and re-driving dead-lettered jobs.
